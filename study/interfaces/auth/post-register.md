# 注册接口学习笔记

## 1. 接口用途

- 注册接口用于创建普通用户账号，并在注册成功后直接返回登录态，减少前端额外再调一次登录接口的成本。
- 当前项目的注册默认创建 `USER` 角色，不允许由前端直接指定角色。

## 2. 基本信息

- 路由：`POST /api/auth/register`
- 请求方法：`POST`
- 控制器入口：`src/modules/auth/auth.controller.ts` 中的 `register`
- Service 入口：`src/modules/auth/auth.service.ts` 中的 `register`

## 3. 请求链路

- 请求先进入 `AuthController.register`。
- `RegisterDto` 校验邮箱、密码和确认密码的基础格式。
- `AuthService.register` 先校验两次密码是否一致。
- 随后调用 `UsersService.register` 创建用户，默认写入 `USER` 角色。
- 用户创建成功后，走与登录一致的 `buildAuthPayload` 流程，直接返回 token 与用户信息。

## 4. DTO 与参数校验知识点

- `RegisterDto` 和登录 DTO 的区别在于多了 `confirmPassword`。
- DTO 负责基础格式校验，业务一致性校验由 Service 负责，例如“两次密码是否一致”。
- 这种分层方式的好处是：
  - DTO 负责静态规则
  - Service 负责业务规则

## 5. 鉴权与权限知识点

- 注册接口也是 `@Public()`，因为用户在注册前通常没有登录态。
- 注册接口创建出来的账号角色固定为 `USER`，这样可以避免前端通过参数伪造管理员权限。
- 注册成功后返回的项目列表，会根据 `USER` 角色过滤出允许访问的前端项目。

## 6. 数据层知识点

- `UsersService.register` 最终复用了统一的用户创建逻辑。
- 创建用户前会先查重邮箱，避免重复注册。
- 用户密码在入库前会统一经过哈希处理。
- 账号创建完成后，refresh token 的哈希也会被立刻持久化，保证后续刷新链路可以工作。

## 7. 响应与异常处理知识点

- 当邮箱已存在时，会抛出冲突类业务异常。
- 当两次密码不一致时，会抛出 400 级别异常。
- 注册接口虽然返回的是认证结果，但最终响应格式仍然由全局拦截器统一包装。

## 8. Swagger 与调试方式

- 可以在 `/api/docs` 中直接调试。
- 调试时建议使用一个新的邮箱，避免命中“邮箱已存在”异常。

## 9. 本接口值得学习的后端设计点

- 注册接口应当属于认证模块，而不是后台管理建人接口。
- 面向终端用户的注册请求不应允许自定义角色。
- 注册成功直接返回登录态，是典型的认证体验优化。
- 注册与登录复用统一发 token 逻辑，可以降低维护成本和行为不一致的风险。
